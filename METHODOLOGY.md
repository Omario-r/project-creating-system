# METHODOLOGY — журнал решений о процессе

> Мета-слой: про то, КАК мы строим, а не про конкретный продукт. К играм и
> проектам отношения не имеет. Читается только при изменении флоу разработки
> (сессия-консультация о процессе). В обычной рабочей сессии не открывается.
>
> Аналог `docs/DECISIONS.md`, но для метода, а не для архитектуры продукта.
>
> **Статус файла: ЧЕРНОВИК.** Оформление журнала и мета-роль «процесс-консультанта»,
> которая его ведёт, проектируются отдельным заходом (см. «Открытые вопросы»).

## Решение 2026-07-13 — Расщепление роли исполнителя (Постановщик / Исполнитель)

### Контекст / проблема
Одна роль «исполнитель» (`AGENT.md`) делает и декомпозицию стори→задачи, и
написание кода. PO хочет: (1) стори самодостаточна для нарезки задач, задача
самодостаточна для написания кода; (2) исполнение задачи можно поручить другому
агенту, подходящему по сложности.

### Решения
1. **Роль исполнителя расщепляется на две:**
   - **Постановщик** (сильный агент) — эпик→стори (Гейт 1) и стори→задачи; пишет
     самодостаточные ТЗ; назначает исполняющего агента; владеет арх-доками
     (`ARCHITECTURE.md`/`DECISIONS.md`); принимает техническую эскалацию.
     Прод-код не пишет.
   - **Исполнитель** (уровень по задаче) — одно ТЗ → код + тесты + локальный
     паспорт модуля + `STATUS.md`. Один коммит-транзакция.

   Имя «Постановщик» выбрано PO: «ставит задачи» = пишет ТЗ, пара к «Исполнителю».
   «Планировщик» отклонён как недоговаривающий (не виден слой архитектуры и
   назначения агентов).

2. **Ось разделения — ответственность, а не модель.** Уровень агента — свойство
   ЗАДАЧИ, назначает Постановщик. Тезис «код = всегда дёшево» отвергнут: сложность
   задачи гуляет, мудрёный алгоритм требует сильной модели даже при идеальном ТЗ.
   Настоящий enabler делегирования — ПОЛНОТА ТЗ, а не дешёвая модель; экономия
   модели — следствие хорошего контракта задачи.

3. **Гейт нарезки стори→задачи — НЕ вводим.** Нарезка остаётся технической
   операцией (`AGENT.md` п.2). Постановщик коммитит ТЗ задач как артефакт; PO
   может заглянуть, но это не обязательная остановка.

4. **DoD при делегировании — гибрид.** Исполнитель ведёт локальный паспорт
   затронутого модуля + `STATUS.md`; Постановщик — `ARCHITECTURE.md`/`DECISIONS.md`.
   Механика без раскола коммита: Постановщик, когда режет стори, ЗАРАНЕЕ пишет
   арх-дельту прямо в ТЗ задачи (он уже знает, что задача трогает карту модулей —
   сам назначил границу); Исполнитель применяет её как есть в том же единственном
   коммите, судительность слабого агента не требуется. Граница «локальное vs
   архитектурное» = существующие триггеры DoD (п.5.3 — добавлен/удалён модуль или
   связь; п.5.4 — выбор из альтернатив с долгими последствиями). Непредвиденное
   арх-влияние → эскалация Исполнитель→Постановщик, не сочинять самому.

5. **Новый канал эскалации: Исполнитель → Постановщик** (техническое: ТЗ неполно,
   нарезка кривая, полез 4-й модуль, непредвиденная архитектура). НЕ доходит до
   PO — иначе разделение съедает время PO. Эскалация к PO остаётся только
   продуктовой (`AGENT.md` п.2, п.4).

6. **Ростер** — каталог агентных ролей, между которыми Постановщик распределяет
   задачи (поле «рекомендованный агент» в ТЗ). Старт минимальный: Постановщик +
   Исполнитель + флаг «тяжёлая» (атрибут задачи, не отдельный архетип — чтобы не
   плодить роли раньше потребности). Пока — СЕКЦИЯ в файле роли Постановщика, не
   отдельный файл. Отдельным файлом станет в Фазе 3 вместе с конкретными
   субагентами и таблицей «уровень → модель».

7. **Два слоя документации разведены явно:**
   - Продуктовый (игра/проект): VISION, EPIC, STORY, ARCHITECTURE, DECISIONS, STATUS.
   - Слой метода (система): CLAUDE/AGENT/ADVISOR + этот журнал. Живёт в шаблоне,
     не копируется в проекты; читается только при изменении флоу.

## Решение 2026-07-13 — Синхронизация правил шаблон↔проекты (оверлей)

### Контекст / проблема
Мета-цель текущего проекта (игры) — отладить флоу для будущих проектов, поэтому
наработки должны возвращаться в шаблон и доезжать до новых проектов. Правила
(`AGENT.md` и пр.) сейчас дублируются между шаблоном и игрой руками; при
расщеплении роли файлов станет больше, рассинхрон — реальный риск. Диагноз:
правила ~95% генерик / ~5% проектные адаптации (по диффу игра↔шаблон различия
ровно в 4 местах: `src`↔`lib`, зависимость pubspec, `flutter analyze` в DoD,
блок веб-таргета), и адаптации вписаны ВНУТРЬ генерик-файлов → синхронизация
обречена быть слиянием с конфликтами. Способ доставки вторичен; первично —
чистое разделение слоёв.

### Решения
1. **Оверлей-модель.** Генерик-правила держатся генериком и копируются в проект
   дословно, без локальных правок. Все проектные адаптации — в отдельном
   overlay-файле `PROJECT.md` (корень проекта): каталог кода (`lib`/`src`),
   команда линтера/гейта качества, платформа, проектные карве-ауты правил (напр.
   блок веб-таргета игры). DoD генерикизуется: не «`flutter analyze` = 0», а
   «линтер проекта = 0», а какой именно — в оверлее.

2. **Физическая граница — папка `system/`.** Генерик-правила живут в отдельной
   папке `system/` (в шаблоне и в каждом проекте); синхронизация вниз = копия
   одной папки, манифест не нужен, граница однозначна. `CLAUDE.md` остаётся в
   корне (Claude Code грузит его автоматически) — тонкий, указывает на `system/`
   и `PROJECT.md`.

3. **Протокол чтения.** Агент читает «генерик (`system/`) + оверлей
   (`PROJECT.md`)». Правка в `CLAUDE.md` (маршрут) и `AGENT.md` п.1 (порядок
   чтения контекста).

4. **Доставка вниз — агентная дословная копия, без внешних инструментов.**
   subtree/сабмодуль/генератор (copier) отвергнуты как машинерия против
   низкотехнологичного духа системы. Синхронизация = команда агенту
   «синхронизируй правила из шаблона»: агент перезаписывает `system/` из шаблона
   (по git-URL, чтобы не зависеть от локального пути на машине) и обновляет штамп.

5. **Прослеживаемость — штамп в `PROJECT.md`:** «синхронизировано с шаблоном @
   `<commit>` (`<дата>`)». Сравнение со `HEAD` шаблона показывает, отстал проект
   или нет. Наработки не только сохранены в шаблоне, но и видно, докуда доехали.

6. **Апстрим наработок — round-trip через шаблон.** Генерик-улучшение, найденное
   в проекте, вносится В ШАБЛОН (`system/`), «почему» — в этот журнал, затем
   синхронизируется вниз. Проект генерик-файлы локально не редактирует.

## Решение 2026-07-13 — Проверка полноты ТЗ (тиражирование по риску)

### Контекст / проблема
Вся модель делегирования держится на полноте ТЗ: неполное ТЗ → Исполнитель
проваливается или возвращает. Кто проверяет полноту стори/ТЗ до передачи?

### Решения
1. **Самопроверка автора слепа к смысловым пробелам.** Постановщик держит контекст
   в голове; именно то, что не доехало до ТЗ, при самоперечитывании он не замечает.
   Самопроверка ловит форму (пустые поля), не смысл — смысловые пробелы видят
   только свежие глаза.
2. **Три уровня по возрастанию цены:**
   - *Универсально, дёшево (сейчас):* самопроверка Постановщика на форму перед
     коммитом нарезки — зеркало того, как Исполнитель проходит критерии перед
     Гейтом 2. → `POSTANOVSHCHIK.md` § 3.
   - *Универсально, реактивно (в дизайне):* эскалация Исполнитель→Постановщик —
     финальный бэкстоп.
   - *Сложные случаи (Фаза 3):* холодная вычитка — дешёвый свежий агент читает ТЗ
     вслепую и возвращает список пробелов ДО запуска Исполнителя. Оправдана там,
     где реактивный round-trip дорог. Триггеры: флаг «тяжёлая», хрупкий домен
     (деньги/данные/расчёты/парсинг), задача задевает архитектуру, крупная стори.
3. Холодная вычитка — субагентная операция, её место — Фаза 3.

## Решение 2026-07-13 — Агент коммитит (стандарт)

Правило «коммитит только PO» (старый `ADVISOR.md` п.7 в игре) опиралось на
проблему `.lock`-файлов в примонтированной песочнице. В Claude Code её нет —
**стандарт: агент коммитит сам** (последний пункт DoD у Исполнителя; конец сессии
у Советника и Постановщика). Шаблонный `ADVISOR.md` это уже отражал; игра
приведена к нему синхронизацией.

## Решение 2026-07-13 — Ростер: две оси, Claude-субагенты, внешние бэкенды (Фаза 3)

1. **Ростер — две оси:** роль (функция) × бэкенд (движок). «Рекомендованный
   агент» в ТЗ = роль + бэкенд.
2. **Claude-бэкенды — субагенты `.claude/agents/`**, тир→модель: Постановщик/
   Советник/«тяжёлая» → Opus, Исполнитель → Sonnet, Критик → Haiku. Исполнителя и
   Критика Постановщик/PO спавнит ЯВНО, по команде (не автопилот) — гейты и «одна
   стори за раз» целы.
3. **Внешние бэкенды** — для overflow (лимиты Claude) и параллели. Механизм
   зависит от поверхности агента: нативно (Claude Task), Bash (CLI-агенты), ручной
   хендофф (GUI — Cursor Composer 2.5 fast, первая запись). Универсальный хендофф
   обеспечивает самодостаточное ТЗ (Фаза 1): задачу можно отдать любому бэкенду,
   результат возвращается коммитом по DoD.
4. `.claude/agents/` — конфиг проекта (не в `system/`), но содержимое генерик
   (читает роль из `system/` + `PROJECT.md`); шаблон несёт канонический набор.

## Решение 2026-07-13 — Верификация: детерминированное у Исполнителя, визуальное у оркестратора

Исполнитель (в т.ч. субагент) закрывает только детерминированное: линтер + тесты.
Визуальную и поведенческую проверку (внутренний браузер/Preview, скриншот,
«читается как стиль») делает оркестратор (у него Browser-инструменты) и PO на
Гейте 2. Субагенту-исполнителю браузерные/computer-use инструменты **не давать** —
иначе при визуальном критерии в ТЗ он тянется к Chrome. Повод: обкатка T1 —
`ispolnitel` без внутреннего браузера запросил Chrome под скриншот-критерий.

Браузерная/визуальная проверка — **один раз на уровне стори (Гейт 2), НЕ после
каждой задачи**: лишние прогоны браузера жгут токены. По задачам — только
детерминированное (линтер/тесты).

## Решение 2026-07-14 — Gigacode в ростере: HANDOFF-контракт и трёхступенчатая приёмка

Консультация PO + Советник (специализация LLM). Основа — обкатка Gigacode на
gotovnost/01 T2 (см. «зерно» в открытых вопросах от 2026-07-13, теперь закрыто).

### Контекст / проблема
Gigacode бесплатен и по полному ТЗ пишет структурно верный код, но: (а) роняет
процессные обязанности (коммит-транзакция, файл стори, запрошенный тест);
(б) пропускает связки в коде (`shouldRepaint`); (в) проверка результата
оркестратором на Opus сожгла ~30% дневной нормы токенов — экономия на коде
разоряется на приёмке.

### Решения
1. **Gigacode — запись в ростере, допуск узкий:** слабый тир, бесплатен, GUI
   (ручной хендофф). Только простые задачи Исполнителя при полном ТЗ; НЕ
   «тяжёлая», НЕ хрупкий домен, НЕ выбор из альтернатив. Ценность: overflow и
   параллель; экономия на рутине — бонус, работающий только с дешёвой приёмкой.
   В ростере пометка о конфиденциальности: внешний сервис видит код.
2. **Контракт сдачи — `system/HANDOFF.md`, а не «скормить AGENT.md».** Причины
   (LLM-специфика): бюджет инструкций слабой модели (чек-лист 5–9 пунктов
   исполняется, 130 строк правил — нет), половина AGENT.md для внешнего агента
   шум (гейты, эскалации, роли), по перекрёстным ссылкам слабая модель не ходит.
   HANDOFF = дистиллят DoD нумерованным чек-листом + канонический шаблон
   промпта-запуска («прочитай HANDOFF + выполни T_N из стори»). Заполненную
   копию промпта Постановщик вписывает прямо в ТЗ задачи (поле «Промпт-запуск»
   в шаблоне задачи) — PO копирует в чат агента без ручных подстановок.
   Фолбэк, если поверхность агента не открывает файлы: вставить содержимое.
3. **ТЗ для внешнего бэкенда строже** (`POSTANOVSHCHIK.md` § 4): точные команды
   линтера/тестов прямо в ТЗ (агент не видит `PROJECT.md`), позитивные списки
   файлов вместо запретов (слабые модели хуже исполняют «не трогай»), пути и
   сигнатуры без отсылок.
4. **Приёмка — три ступени по возрастанию цены** (`ROSTER.md`), урок T2 «чинил
   Opus» не повторять: (1) ре-ран линтера/тестов — Bash, токен-копейки;
   (2) пост-критик: `critic` (Haiku) получает В ПРОМПТ ТЗ + `git diff` — дизайн
   «критик не видит дифф» решён вставкой диффа, Bash критику не нужен;
   (3) находки возвращаются внешнему агенту (бесплатному) на исправление;
   доводка сильной моделью — только после двух неудачных возвратов (и это
   сигнал сузить допуск). Полный аудит оркестратором — только разовая обкатка
   нового бэкенда.
5. **Судительность промптом не лечится:** пропущенные связки ловит только
   внешняя проверка — приёмка не опция, а третья несущая часть контракта
   (ТЗ = что делать, HANDOFF = как сдавать, приёмка = страховка).
6. **Приёмка закреплена за Постановщиком, роль «оркестратор» не вводится.**
   PO заметил дыру: в правилах слово «оркестратор» есть, а роли и команды в
   маршрутизаторе — нет. Решение: приёмка результата внешней задачи — третья
   операция Постановщика (`прими результат задачи <N>`, маршрут в `CLAUDE.md`);
   он назначил бэкенд и владеет ТЗ — ему и принимать. Код при приёмке он не
   чинит (два неудачных возврата подряд → эскалация PO). «Оркестратор» в
   остальных местах правил = главная сессия Claude Code (спавн субагентов,
   браузерная проверка на Гейте 2), не отдельная роль.

## Решение 2026-07-18 — Слой дизайна: роль Дизайнера, канон визуала в репо, мокапы как предусловие декомпозиции

### Контекст / проблема
Система передаёт поведение (критерии «сделай — увидишь»), но не визуальный
замысел — он не выживает при передаче Постановщик → Исполнитель. Обкатка на
игре: (а) референс «Ронин v4» жил ВНЕ репо — один агент изобрёл свою палитру
(попытка откачена PO), другой ошибочно считал файл недоступным; (б) у визуала
не было ни документа, ни владельца (аналога `ARCHITECTURE.md`/Постановщика);
(в) визуальная проверка — поздняя (Гейт 2) и дорогая. Запрос PO: визуал каждого
экрана, «чтобы было понятно, что и как должно выглядеть».

### Решения
1. **Канон визуала — артефакт в репо.** `docs/DESIGN.md` (подход, токены
   «имя → значение → где в коде», язык форм, отступления от конвенций, индекс
   экранов со статусами) + `docs/design/<экран>.html` — **один файл = один
   экран** (решение PO; скринбук не сущность — страница-индекс по мере нужды).
   Референсы тоже кладутся в репо. Каркасы — `system/templates/DESIGN.md` и
   `system/templates/screen-mockup.html` (канон только в блоке токенов,
   остальное — иллюстрация).
2. **Формат мокапа — HTML.** Единственный формат «картинка для PO (браузер) +
   спека для агента (точные токены)»; агент сам генерит и итерирует в диалоге,
   git-friendly. Проверено боем («Ронин v4»: палитра, взятая из HSL-констант
   файла, закрыла инцидент). Figma не в каноне (агент не итерирует, канон вне
   репо, round-trip дорог) — допустима как личный черновик PO с переводом в
   HTML; картинки (токены не извлечь), текстовые wireframes (не передают
   визуальный язык), golden-скриншоты (верификация «после», не замысел «до»)
   отклонены. Веб-рендер ≠ рендер платформы — принято: мокап задаёт
   направление, не копию 1:1.
3. **Отдельная роль Дизайнер** (`system/DIZAINER.md`, тир сильный/Opus), а не
   сценарий Советника: (а) канону нужен владелец, а Советник по природе ничем
   не владеет — его выход черновики; (б) UX/UI — отдельный корпус знаний,
   в ADVISOR.md он грузился бы в каждую консультацию против атомарности чтения;
   (в) прецедент — расщепление Постановщик/Исполнитель по оси ответственности.
   Владение: `DESIGN.md` + `docs/design/`; Постановщик ссылается, Исполнитель
   читает, изменение визуала — только через Дизайнера.
4. **Нюансы конвенций в файл роли НЕ выписываются.** Любой список неполон и
   устареет; знания приносит модель сильного тира, файл кодирует протокол:
   платформа × класс приложения (обе оси — `PROJECT.md`, секция «Дизайн») →
   режим конвенций (стандартное приложение = платформенные паттерны по
   умолчанию, отступления явно; игра = кастомный язык, конвенции только в
   системной зоне: safe area, жесты, шторки) → **конвенция-чек** перед
   утверждением: отступление не запрещено, но не молча. Источник — опыт PO на
   другом проекте («веб-паттерны в мобилке», таб-бар: лид ловил нарушения
   глазами — эту функцию и выполняет чек).
5. **Дизайн — предусловие декомпозиции, весь эпик сразу** (решение PO: строго,
   на уровне эпика). Эпик с видимым UI не режется, пока все экраны из секции
   «Затронутые экраны» EPIC.md не «утверждён PO» — аналогия «работа не
   начинается, пока открытые вопросы не пусты». Нового формального гейта нет.
   Инцидент с палитрой = работа началась до фиксации канона; предусловие
   закрывает это по построению.
6. **Передача вниз по цепочке:** поле «Дизайн-референс» в ТЗ (зеркало
   арх-дельты): мокап + токены; визуальный критерий без ссылки = неполное ТЗ
   (самопроверка Постановщика § 3). Исполнителю: значения ТОЛЬКО из референса,
   нет/расходится → эскалация, свою палитру не изобретать (AGENT.md п. 3, п. 4).
   Внешним бэкендам токены дублируются числами прямо в ТЗ + строка в чек-листе
   HANDOFF (файлы проекта они не читают — POSTANOVSHCHIK § 4).

## План внедрения (фазами; каждая — ×2: шаблон + проект, в шаблоне обобщённо)

Рекомендация: начинать с Фазы 1 — пока контракты не строгие, разделять роли
опасно (риск «протекающих контрактов»).

- **Фаза 1 — Контракты (фундамент, низкий риск).** Обогатить `STORY.md`
  (самодостаточная стори) + под-шаблон задачи (всё для кода + поле «агент» +
  инлайн-чек-лист DoD + слот арх-дельты). Поведение ещё старое, меняются шаблоны.
- **Фаза 2 — Роли + структура синхронизации.** `AGENT.md` → `AGENT.md`
  (Исполнитель) + новый файл роли Постановщика; гибрид-DoD и обе эскалации.
  Здесь же оверлей: перенос генерик-правил в `system/`, создание `PROJECT.md`,
  генерикизация DoD, правка протокола чтения (`CLAUDE.md`, `AGENT.md` п.1).
  Плюс триггер возврата Гейта 3 (по аналогии с Гейтом 2). Фаза объёмная — при
  исполнении может резаться на подзадачи по общим правилам.
- **Фаза 3 — Ростер + делегирование.** Ростер отдельным файлом (генерик, в
  `system/`); конкретные субагенты Claude Code (`.claude/agents/`, модель на
  агента); привязка поля «рекомендованный агент» к реальному запуску. Плюс
  холодная вычитка ТЗ для сложных случаев (см. решение о проверке полноты).
