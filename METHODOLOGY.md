# METHODOLOGY — журнал решений о процессе

> Мета-слой: про то, КАК мы строим, а не про конкретный продукт. К играм и
> проектам отношения не имеет. Читается только при изменении флоу разработки
> (сессия-консультация о процессе). В обычной рабочей сессии не открывается.
>
> Аналог `docs/DECISIONS.md`, но для метода, а не для архитектуры продукта.
>
> **Статус файла: ЧЕРНОВИК.** Оформление журнала и мета-роль «процесс-консультанта»,
> которая его ведёт, проектируются отдельным заходом (см. «Открытые вопросы»).

## Решение 2026-07-13 — Расщепление роли исполнителя (Постановщик / Исполнитель)

### Контекст / проблема
Одна роль «исполнитель» (`AGENT.md`) делает и декомпозицию стори→задачи, и
написание кода. PO хочет: (1) стори самодостаточна для нарезки задач, задача
самодостаточна для написания кода; (2) исполнение задачи можно поручить другому
агенту, подходящему по сложности.

### Решения
1. **Роль исполнителя расщепляется на две:**
   - **Постановщик** (сильный агент) — эпик→стори (Гейт 1) и стори→задачи; пишет
     самодостаточные ТЗ; назначает исполняющего агента; владеет арх-доками
     (`ARCHITECTURE.md`/`DECISIONS.md`); принимает техническую эскалацию.
     Прод-код не пишет.
   - **Исполнитель** (уровень по задаче) — одно ТЗ → код + тесты + локальный
     паспорт модуля + `STATUS.md`. Один коммит-транзакция.

   Имя «Постановщик» выбрано PO: «ставит задачи» = пишет ТЗ, пара к «Исполнителю».
   «Планировщик» отклонён как недоговаривающий (не виден слой архитектуры и
   назначения агентов).

2. **Ось разделения — ответственность, а не модель.** Уровень агента — свойство
   ЗАДАЧИ, назначает Постановщик. Тезис «код = всегда дёшево» отвергнут: сложность
   задачи гуляет, мудрёный алгоритм требует сильной модели даже при идеальном ТЗ.
   Настоящий enabler делегирования — ПОЛНОТА ТЗ, а не дешёвая модель; экономия
   модели — следствие хорошего контракта задачи.

3. **Гейт нарезки стори→задачи — НЕ вводим.** Нарезка остаётся технической
   операцией (`AGENT.md` п.2). Постановщик коммитит ТЗ задач как артефакт; PO
   может заглянуть, но это не обязательная остановка.

4. **DoD при делегировании — гибрид.** Исполнитель ведёт локальный паспорт
   затронутого модуля + `STATUS.md`; Постановщик — `ARCHITECTURE.md`/`DECISIONS.md`.
   Механика без раскола коммита: Постановщик, когда режет стори, ЗАРАНЕЕ пишет
   арх-дельту прямо в ТЗ задачи (он уже знает, что задача трогает карту модулей —
   сам назначил границу); Исполнитель применяет её как есть в том же единственном
   коммите, судительность слабого агента не требуется. Граница «локальное vs
   архитектурное» = существующие триггеры DoD (п.5.3 — добавлен/удалён модуль или
   связь; п.5.4 — выбор из альтернатив с долгими последствиями). Непредвиденное
   арх-влияние → эскалация Исполнитель→Постановщик, не сочинять самому.

5. **Новый канал эскалации: Исполнитель → Постановщик** (техническое: ТЗ неполно,
   нарезка кривая, полез 4-й модуль, непредвиденная архитектура). НЕ доходит до
   PO — иначе разделение съедает время PO. Эскалация к PO остаётся только
   продуктовой (`AGENT.md` п.2, п.4).

6. **Ростер** — каталог агентных ролей, между которыми Постановщик распределяет
   задачи (поле «рекомендованный агент» в ТЗ). Старт минимальный: Постановщик +
   Исполнитель + флаг «тяжёлая» (атрибут задачи, не отдельный архетип — чтобы не
   плодить роли раньше потребности). Пока — СЕКЦИЯ в файле роли Постановщика, не
   отдельный файл. Отдельным файлом станет в Фазе 3 вместе с конкретными
   субагентами и таблицей «уровень → модель».

7. **Два слоя документации разведены явно:**
   - Продуктовый (игра/проект): VISION, EPIC, STORY, ARCHITECTURE, DECISIONS, STATUS.
   - Слой метода (система): CLAUDE/AGENT/ADVISOR + этот журнал. Живёт в шаблоне,
     не копируется в проекты; читается только при изменении флоу.

## Решение 2026-07-13 — Синхронизация правил шаблон↔проекты (оверлей)

### Контекст / проблема
Мета-цель текущего проекта (игры) — отладить флоу для будущих проектов, поэтому
наработки должны возвращаться в шаблон и доезжать до новых проектов. Правила
(`AGENT.md` и пр.) сейчас дублируются между шаблоном и игрой руками; при
расщеплении роли файлов станет больше, рассинхрон — реальный риск. Диагноз:
правила ~95% генерик / ~5% проектные адаптации (по диффу игра↔шаблон различия
ровно в 4 местах: `src`↔`lib`, зависимость pubspec, `flutter analyze` в DoD,
блок веб-таргета), и адаптации вписаны ВНУТРЬ генерик-файлов → синхронизация
обречена быть слиянием с конфликтами. Способ доставки вторичен; первично —
чистое разделение слоёв.

### Решения
1. **Оверлей-модель.** Генерик-правила держатся генериком и копируются в проект
   дословно, без локальных правок. Все проектные адаптации — в отдельном
   overlay-файле `PROJECT.md` (корень проекта): каталог кода (`lib`/`src`),
   команда линтера/гейта качества, платформа, проектные карве-ауты правил (напр.
   блок веб-таргета игры). DoD генерикизуется: не «`flutter analyze` = 0», а
   «линтер проекта = 0», а какой именно — в оверлее.

2. **Физическая граница — папка `system/`.** Генерик-правила живут в отдельной
   папке `system/` (в шаблоне и в каждом проекте); синхронизация вниз = копия
   одной папки, манифест не нужен, граница однозначна. `CLAUDE.md` остаётся в
   корне (Claude Code грузит его автоматически) — тонкий, указывает на `system/`
   и `PROJECT.md`.

3. **Протокол чтения.** Агент читает «генерик (`system/`) + оверлей
   (`PROJECT.md`)». Правка в `CLAUDE.md` (маршрут) и `AGENT.md` п.1 (порядок
   чтения контекста).

4. **Доставка вниз — агентная дословная копия, без внешних инструментов.**
   subtree/сабмодуль/генератор (copier) отвергнуты как машинерия против
   низкотехнологичного духа системы. Синхронизация = команда агенту
   «синхронизируй правила из шаблона»: агент перезаписывает `system/` из шаблона
   (по git-URL, чтобы не зависеть от локального пути на машине) и обновляет штамп.

5. **Прослеживаемость — штамп в `PROJECT.md`:** «синхронизировано с шаблоном @
   `<commit>` (`<дата>`)». Сравнение со `HEAD` шаблона показывает, отстал проект
   или нет. Наработки не только сохранены в шаблоне, но и видно, докуда доехали.

6. **Апстрим наработок — round-trip через шаблон.** Генерик-улучшение, найденное
   в проекте, вносится В ШАБЛОН (`system/`), «почему» — в этот журнал, затем
   синхронизируется вниз. Проект генерик-файлы локально не редактирует.

## План внедрения (фазами; каждая — ×2: шаблон + проект, в шаблоне обобщённо)

Рекомендация: начинать с Фазы 1 — пока контракты не строгие, разделять роли
опасно (риск «протекающих контрактов»).

- **Фаза 1 — Контракты (фундамент, низкий риск).** Обогатить `STORY.md`
  (самодостаточная стори) + под-шаблон задачи (всё для кода + поле «агент» +
  инлайн-чек-лист DoD + слот арх-дельты). Поведение ещё старое, меняются шаблоны.
- **Фаза 2 — Роли + структура синхронизации.** `AGENT.md` → `AGENT.md`
  (Исполнитель) + новый файл роли Постановщика; гибрид-DoD и обе эскалации.
  Здесь же оверлей: перенос генерик-правил в `system/`, создание `PROJECT.md`,
  генерикизация DoD, правка протокола чтения (`CLAUDE.md`, `AGENT.md` п.1).
  Плюс триггер возврата Гейта 3 (по аналогии с Гейтом 2). Фаза объёмная — при
  исполнении может резаться на подзадачи по общим правилам.
- **Фаза 3 — Ростер + делегирование.** Ростер отдельным файлом (генерик, в
  `system/`); конкретные субагенты Claude Code (`.claude/agents/`, модель на
  агента); привязка поля «рекомендованный агент» к реальному запуску.

## Открытые вопросы (следующие заходы)

- Мета-роль «консультация о процессе» + финальный формат и имя этого журнала.
- Параллельные исполнители внутри одной стори для независимых задач — надо ли и
  какой ценой (сейчас правило: одна стори за раз, параллель — только решением PO).
- Правило «коммитит только PO» (`ADVISOR.md` п.7) опиралось на проблему
  `.lock`-файлов в примонтированной песочнице. В Claude Code этой проблемы, по
  словам PO, нет — правило кандидат на отмену/ревизию (агент коммитит сам).
