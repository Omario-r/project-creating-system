# AGENT.md — Правила работы агента в этом проекте

> Этот файл читается агентом в начале КАЖДОЙ сессии. Он — единственный обязательный
> вход. Всё остальное читается по необходимости согласно правилам ниже.

## 0. Роли

- **Product Owner (PO)** — человек. Владеет Vision, эпиками, приоритетами. Принимает работу.
- **Агент** — ты. Декомпозируешь эпики в стори, стори в задачи, реализуешь, поддерживаешь память проекта.

## 1. Порядок чтения контекста (атомарность)

Читай минимум, необходимый для текущей работы. Никогда не читай весь проект.

| Тип работы | Что читать |
|---|---|
| Декомпозиция эпика | `docs/VISION.md` → `docs/epics/<эпик>/EPIC.md` → `docs/ARCHITECTURE.md` |
| Нарезка стори в задачи | Стори → паспорта затронутых модулей (`src/<модуль>/README.md`) |
| Выполнение задачи | Задача → стори (для контекста) → паспорта модулей из секции «Затронутые модули» |
| Архитектурный вопрос | `docs/ARCHITECTURE.md` → `docs/DECISIONS.md` (поиск по теме, не целиком) |

**Правило 2–3 модулей:** если для выполнения задачи нужно прочитать более трёх
паспортов модулей — остановись. Это сигнал, что задача нарезана неверно или
архитектура слишком связана. Эскалируй PO (см. п. 4).

## 2. Граница решений: техническое vs продуктовое

- **Техническое** (решаешь сам): выбор структуры данных, внутреннее устройство модуля,
  именование, рефакторинг без изменения интерфейсов, выбор алгоритма.
- **Продуктовое** (только эскалация PO): всё, что меняет поведение, видимое пользователю;
  границы стори; приоритеты; отказ от критерия приёмки; добавление функциональности,
  которой нет в стори.

**Тест:** «заметит ли пользователь разницу?» Если да или не уверен — это продуктовый
вопрос. Запиши его в секцию «Открытые вопросы к PO» стори и не начинай работу по
этой части до ответа.

## 3. Правила декомпозиции

### Эпик → Стори
- Каждая стори — законченная ценность: «пользователь может X, чтобы Y».
- У стори есть проверяемые критерии приёмки (конечный список).
- Стори имеет явные границы: секция «Не входит» обязательна.
- Сумма стори покрывает цель эпика; ничего лишнего сверх границ эпика.
- Результат декомпозиции — список стори на утверждение PO (**Гейт 1**). Работа не
  начинается до утверждения.

### Стори → Задачи
- Задача выполняется за одну сессию (ориентир: одна связная правка,
  ~≤300 строк диффа). Оценка на глаз при нарезке; если по ходу дифф явно
  перерастает ориентир — остановись: задача нарезана неверно, заверши
  связную часть и перенарежь остаток.
- Задача затрагивает не более 2–3 модулей.
- У задачи есть проверяемый результат (что можно запустить/проверить).
- Порядок задач учитывает зависимости; каждая задача оставляет проект в рабочем состоянии.

## 4. Эскалация

Эскалируй PO (останови работу по фрагменту, зафиксируй вопрос), если:
- вопрос продуктовый (п. 2);
- задачу невозможно нарезать по правилам п. 3 — стори слишком большая;
- нарушено правило 2–3 модулей — архитектура сопротивляется;
- критерии приёмки противоречат друг другу или Vision.

Формат: запись в секции «Открытые вопросы к PO» соответствующей стори + краткое
описание вариантов с твоей рекомендацией.

**Неблокирующие находки** (идея, «было бы неплохо», баг вне текущей задачи) —
это не эскалация и не повод расширять скоуп: одна строка в `docs/BACKLOG.md`,
работа продолжается. Критичный баг — дополнительно в STATUS.md («Требуется от PO»).

## 5. Definition of Done (для каждой задачи)

Задача НЕ считается выполненной, пока не сделано всё:

1. Код написан, проверяемый результат достигнут, существующие тесты проходят.
2. Обновлены паспорта модулей (`src/<модуль>/README.md`), если менялся интерфейс,
   зависимости или инварианты.
3. Обновлён `docs/ARCHITECTURE.md`, если появился/удалён модуль или связь между модулями.
4. Записано решение в `docs/DECISIONS.md`, если было принято архитектурное решение
   (выбор из альтернатив с долгосрочными последствиями).
5. Статусы обновлены: задачи — в файле стори; сменился статус стори —
   также в таблице EPIC.md.
6. Обновлён `docs/STATUS.md` — дашборд PO: что в работе, что требуется от PO,
   последние результаты.
7. Сделан коммит задачи (см. п. 8). Это последнее действие каждой сессии,
   без исключений.

> Помни: документация — не описание кода, а интерфейс между сессиями. Следующий
> агент придёт с нулевым контекстом. Пиши для него.

## 6. Гейты (контрольные точки PO)

- **Гейт 1 — Приёмка декомпозиции.** Список стори эпика утверждается PO до начала
  работы. После утверждения переведи эпик «в работе» (EPIC.md и ROADMAP.md).
- **Гейт 2 — Приёмка стори.** Стори закрывается PO по критериям приёмки.
  Перед сдачей сам пройди каждый критерий и проставь галочку с доказательством
  («запустил X → увидел Y») — без доказательств стори на приёмку не выносится.
  Если PO возвращает стори: до начала исправлений переведи статус в
  «возвращена (Гейт 2)», преврати замечания PO в новые задачи (пометка
  «возврат»), сними галочки с непройденных критериев. Дальше обычный цикл.
- **Гейт 3 — Ревью эпика.** После закрытия всех стори PO проводит ревью эпика.
  Итоги — в секцию «Ревью» EPIC.md; 1–2 урока, важных для будущих эпиков, —
  в ROADMAP.md («Уроки»). Статус эпика в ROADMAP.md — «закрыт».

Агент не проходит гейты самостоятельно. Дошёл до гейта — остановись и передай PO.

В работе одновременно ровно одна стори. Параллельный второй исполнитель —
только по явному решению PO.

## 7. Паспорта модулей

Каждый модуль в `src/` обязан иметь `README.md` по шаблону `docs/templates/MODULE.md`
(30–50 строк). Создание модуля без паспорта запрещено. Паспорт — единственное, что
другие агенты будут знать о модуле, не читая код.

## 8. Git: коммит = транзакция

- Вся работа в main. Ветка — только по явному решению PO (эксперимент/спайк).
- После каждой задачи — один коммит: код + все обновлённые документы
  (паспорта, стори, STATUS.md) вместе. Посреди задачи не коммитить.
- Сообщение: `<эпик>/<стори> T<N>: <суть>`.
- Коммит — последний пункт DoD (после обновления STATUS.md).

## 9. Возобновление после обрыва

Первое действие любой сессии — `git status`:

- Чисто → последний коммит согласован с документами, работай как обычно.
- Есть незакоммиченные правки → прошлая сессия оборвалась. Истина — последний
  коммит. Оцени дифф:
  - понятный и почти завершённый → доделай, закрой как обычную задачу;
  - мутный или большой → откати к последнему коммиту, задача делается заново.
  - В обоих случаях зафиксируй случившееся в «Заметках по выполнению» задачи.

## 10. Режим качества

Задан в VISION.md («Принципы и ограничения»). Не указан — спроси PO, не выбирай сам.

Безусловно, в любом режиме:
- Смоук-тест: с первой стори в проекте живёт минимальный тест «проект
  запускается, основной сценарий проходит без падения». Поддерживается актуальным.
- Существующие тесты проходят перед коммитом каждой задачи.

Режимы:
- **прототип** — тесты по умолчанию не пишутся, НО при декомпозиции агент
  обязан помечать хрупкие места (деньги, данные, расчёты, парсинг) и вставлять
  тест-задачи с пометкой «рекомендован» — PO оставляет или вычёркивает их на
  Гейте 1. Хрупкость, найденная посреди работы, — строка в BACKLOG.
- **продукт** — каждая стори получает критерий «ключевое поведение покрыто
  автотестом» и задачи на тесты. Гейт 2 не проходится с красными тестами.
