# Шаблон проекта: агентная разработка (PO + агенты)

Универсальный каркас для разработки ПО с помощью ИИ-агентов, где человек выступает
в роли Product Owner. Клонируй эту папку в начале каждого нового проекта.

Это полный документ системы — читается здесь, в шаблоне. В живом проекте
единственный PO-документ — памятка `system/PAMYATKA.md`: роли, гейты, словарик.

## Идея в одном абзаце

Ты — Product Owner: держишь видение и принимаешь работу. Всё между «чего хотим» и
«готовый код» делают ИИ-агенты по строгим правилам. Твоё внимание сосредоточено в
нескольких контрольных точках — гейтах, — остальное система тащит сама. Документы
здесь не описание кода, а **интерфейс между сессиями**: каждый агент приходит с
нулевым контекстом и восстанавливает его из документов, а не из твоей памяти.

## Роли

Пять агентных ролей плюс ты (PO). Переключает их `CLAUDE.md` по команде:

- **PO (ты)** — видение, приоритеты, приёмка. Единственный, кто проходит гейты.
- **Советник** (`system/ADVISOR.md`) — «консультация»: сырая идея, Vision, эпик.
  Спорит, предлагает варианты с ценой, пишет черновики. Кода не касается.
- **Дизайнер** (`system/DIZAINER.md`) — «дизайн-сессия»: HTML-мокапы экранов +
  DESIGN.md (токены, визуальный язык). Владеет каноном визуала; проверяет мокапы
  по конвенциям платформы, отступления выносит тебе явно. Эпик с видимым UI не
  режется, пока его экраны не утверждены.
- **Постановщик** (`system/POSTANOVSHCHIK.md`) — «декомпозируй эпик» и «нарежь
  стори на задачи». Пишет самодостаточные ТЗ, назначает исполняющего агента,
  ведёт арх-доки. Прод-код не пишет.
- **Исполнитель** (`system/AGENT.md`) — «возьми/выполни задачу». Берёт одно ТЗ и
  доводит до кода. Может быть более простым агентом, чем Постановщик.
- **Методолог** (`system/METODOLOG.md`) — «сессия методолога», «ретро», «аудит
  проекта». Работает над самой системой: уроки → шаблон, гигиена процесса,
  доработка правил. Продукта не касается; его файл другие роли не читают.

## Жизненный цикл проекта

```
сырая идея ──консультация──► «не делать» (успех: идея умерла за разговор)
                │
                ▼ «делать»
        VISION + режим качества ──► эпик (Советник: цель, границы, очередь)
                                        │
              ┌─────────────────────────┘
              ▼
   Дизайнер: экраны эпика → мокапы → ты утверждаешь (если эпик трогает видимое)
              │
              ▼
   Постановщик: эпик → стори ──► PREVIEW: скользишь глазами, дефолт go
              │
              ▼
   Постановщик: стори → ТЗ задач (артефакт; на каждой задаче — агент)
              │
              ▼
   Исполнитель: ТЗ → код, коммит ──► ГЕЙТ 2: Постановщик принимает стори ×N
              │
              ▼
        ГЕЙТ 3: ревью эпика, уроки → ROADMAP ──► следующий эпик
```

Эпик ведёт Постановщик. Твоя обязательная остановка одна — **Гейт 3** (приёмка
эпика); плюс лёгкое: скользнуть по preview нарезки (дефолт go) и ответить на
вопросы, когда Постановщик вынырнет. Гейт 2 (приёмка стори) он ведёт сам и чейнит
стори до конца эпика. Между Постановщиком и Исполнителем есть свой, технический
канал возврата (неполное ТЗ, кривая нарезка) — он до тебя не доходит. Что именно
смотреть — `system/PAMYATKA.md`.

## Три слоя файлов (понять сразу)

- **Метод** (`system/`, `CLAUDE.md`) — правила, одинаковые во всех проектах.
  Синхронизируются из шаблона, локально их не редактируют.
- **Проект** (`docs/`, код) — содержимое именно этого продукта.
- **Оверлей** (`PROJECT.md`) — тонкая прослойка стыка: стек, каталог кода, линтер.

Улучшил метод — вноси в шаблон и синхронизируй вниз; меняешь продукт — это `docs/`.
Это разделение — то, что позволяет одному шаблону обслуживать много проектов.

```
.claude/

  agents/              #   субагенты: ispolnitel, ispolnitel-heavy, critic
system/                # Генерик-правила — синхронизируются из шаблона, локально не править
  PAMYATKA.md          #   памятка PO: роли, гейты, словарик — единственный PO-док в проекте
  AGENT.md             #   роль Исполнителя (одно ТЗ → код)
  POSTANOVSHCHIK.md    #   роль Постановщика (эпик→стори, стори→ТЗ задач, арх-доки)
  ADVISOR.md           #   роль Советника (идея, Vision, эпик)
  DIZAINER.md          #   роль Дизайнера (мокапы экранов + DESIGN.md, канон визуала)
  METODOLOG.md         #   роль Методолога (доработка системы, ретро, аудит; читается только в своей сессии)
  ROSTER.md            #   каталог ролей × бэкендов, приёмка внешнего агента
  HANDOFF.md           #   контракт сдачи для внешнего агента (чек-лист + промпт)
  templates/
    EPIC.md            #   шаблон эпика
    STORY.md           #   шаблон стори (ТЗ задач живут внутри)

  ROADMAP.md           #   очередь эпиков + уроки
  STATUS.md            #   дашборд состояния (начинай отсюда)
  ARCHITECTURE.md      #   карта модулей
  DECISIONS.md         #   журнал ключевых решений
  BACKLOG.md           #   лоток идей и багов
  DESIGN.md            #   визуальная конституция: токены, индекс экранов (ведёт Дизайнер)
  design/<экран>.html  #   мокапы: один файл = один экран (канон визуала)
  epics/<имя-эпика>/EPIC.md + stories/01-<стори>.md
<каталог кода>/        # Код (каталог задаётся в PROJECT.md; по умолчанию src/)
  <модуль>/README.md   #   паспорт модуля + код
```

## Быстрый старт нового проекта

1. Скопируй папку шаблона, переименуй под проект. Этот README замени на README
   своего продукта — полный документ системы остаётся в шаблоне, рабочая
   шпаргалка (`system/PAMYATKA.md`) едет с проектом.
2. Заполни `PROJECT.md` — стек, каталог кода, команда линтера, платформа.
   Команды линтера/тестов продублируй в allowlist `.claude/settings.json`,
   если их нет в преднаборе, — агенты не будут стопориться на разрешениях.
3. Заполни `docs/VISION.md` — сам или в сессии-консультации
   («Консультация: помоги сформулировать Vision»). Самый важный вопрос —
   «эксперимент или надолго?»: он задаёт режим качества (прототип — быстро,
   без обязательных тестов, но со смоук-тестом; продукт — каждая стори покрыта
   автотестом). Меняется потом только осознанно, через консультацию.
4. Создай первый эпик — сам по шаблону или в консультации.
5. Эпик трогает видимое → «Дизайн-сессия по эпику <имя>» (Дизайнер): мокапы
   затронутых экранов + DESIGN.md, утверди каждый. Без этого декомпозиция
   не начнётся.
6. «Декомпозируй эпик <имя> в стори» (Постановщик) → нарезку вслепую вычитает
   Критик, тебе — **preview** (скользни глазами, дефолт «поехали»).
7. «Нарежь стори <N> на задачи» (Постановщик) → «Возьми стори <N> в работу»
   (Исполнитель). Дальше Постановщик ведёт эпик сам: принимает стори (**Гейт 2**)
   и чейнит до конца.
8. Отвечай на эскалации, когда Постановщик вынырнет (сомнение / продуктовая
   развилка).
9. Все стори закрыты → ревью эпика (**Гейт 3**, твоя приёмка) → следующий эпик.

## Типовые команды агентам

> В Claude Code префикс «Прочитай X.md» не нужен: `CLAUDE.md` маршрутизирует сам
> по команде. Полные команды ниже — для любых других инструментов.

**Консультации (Советник, `system/ADVISOR.md`):**
- `Сырая идея: <две фразы>. Помоги понять, стоит ли делать.`
- `Помоги сформулировать Vision проекта.`
- `Хочу обсудить новый эпик: <идея в двух словах>.`

**Дизайн (Дизайнер, `system/DIZAINER.md`):**
- `Дизайн-сессия по эпику <имя>.` — экраны эпика → мокапы → утверждение.
- `Заведи DESIGN.md: обсудим визуальный подход проекта.`
- `Обнови мокап экрана <X>: <что меняем>.`

**Слой метода (Методолог, `system/METODOLOG.md`):**
- `Сессия методолога: <что обсуждаем в системе>.` — доработка правил/шаблонов.
- `Ретро по <эпику/обкатке>.` — уроки → шаблон → синхронизация.
- `Аудит проекта.` — гигиена процесса: протухшие доки, эпики-зомби, рассинхрон.

**Декомпозиция (Постановщик, `system/POSTANOVSHCHIK.md`):**
- `Декомпозируй эпик <имя> в стори.`
- `Нарежь стори docs/epics/<эпик>/stories/<файл> на задачи.`

**Исполнение (Исполнитель, `system/AGENT.md`)** — три пути запуска, см. ниже:
- `Возьми стори <файл> в работу: начни с первой задачи.` /
  `Продолжи стори <файл> со следующей задачи.`
- `Запусти ispolnitel на стори <N> T<M>.` (субагент из сессии оркестратора)
- `Ответы на открытые вопросы: <...>. Обнови стори и продолжай.`

## Как запускать Исполнителя (3 пути)

У каждой задачи в ТЗ есть «рекомендованный агент» = роль + бэкенд:

1. **Спавн субагента через оркестратора** (по умолчанию для Claude). В сессии
   Постановщика/оркестратора скажи: «запусти `ispolnitel` на стори N T_M».
   Оркестратор поднимает субагента из `.claude/agents/` (модель по тиру) через
   Task-инструмент; тот читает ТЗ + `system/AGENT.md` + `PROJECT.md`, пишет код,
   гоняет линтер/тесты, коммитит и возвращает отчёт. Видишь итог, не каждый шаг.
2. **Отдельная сессия Исполнителя** (полная видимость). Новая сессия Claude Code:
   «Возьми стори N T_M в работу». Модель выбираешь ты при старте сессии (по тиру
   ТЗ). Видишь каждый шаг — когда хочешь контролировать ход.
3. **Внешний бэкенд** (overflow при лимитах Claude / параллель). GUI-агенту
   скопируй в чат готовый промпт-запуск прямо из ТЗ задачи (канонический шаблон —
   `system/HANDOFF.md`); агент с CLI — запуск через Bash. Результат возвращается
   коммитом. Дальше: «прими результат задачи T_N» — Постановщик прогонит три
   дешёвые ступени приёмки (ре-ран линтера/тестов → пост-критик на диффе →
   текст-возврат агенту), протокол — `system/ROSTER.md`.

Критик (холодная вычитка вслепую) — дешёвый агент с тремя режимами: нарезка
(Постановщик гоняет на каждой декомпозиции перед preview), ТЗ (сложные задачи до
запуска Исполнителя), пост-критик (сверка результата внешнего агента). Правила —
`system/CRITIC.md`; запускается явно, как п. 1.

## Сценарии

### Пришла сырая идея

«Консультация: сырая идея — приложение для учёта коллекции винила. Стоит ли?»
Советник не бросится строить — сначала вопросы: чья боль, что эти люди делают
сейчас, какое предположение самое рискованное. Потом предложит самый дешёвый
способ его проверить — часто это не код, а три разговора с людьми. Исхода три:
«делать» — сессия тут же перетекает в Vision; «проверить дешевле» — эксперимент,
иногда прототип с Vision из одной гипотезы; «не делать» — идея умерла за час
вместо трёх недель разработки. Советник обязан спорить: если он поддакивает —
напомни ему п. 1 ADVISOR.md.

### Обычная неделя

**Понедельник.** «Консультация: обсудить эпик — крафт предметов». Советник
сначала разберёт с тобой BACKLOG — за неделю там накопились идеи и пара багов;
что-то станет эпиком, что-то умрёт. Потом цель через ценность, границы
(«не входит» — половина пользы), размер, место в очереди. На выходе — EPIC.md
и строка в ROADMAP.

**Вторник.** Эпик трогает видимое — сначала «Дизайн-сессия по эпику крафт»:
Дизайнер соберёт список затронутых экранов, нарисует HTML-мокапы (смотришь в
браузере, правки — репликой), прогонит их по конвенциям платформы и вынесет
отступления явно. Утверждённые мокапы — канон: без них Постановщик резать не
станет, а Исполнитель потом возьмёт цвета и размеры из них, не из своей головы.

Затем «Декомпозируй эпик крафт в стори». Постановщик прогоняет нарезку через
Критика (вычитка вслепую) и приносит тебе **preview** — список стори. Дефолт
«поехали»: работа стартует, молчание = go. Но скользнуть глазами стоит — это твои
10 минут максимального рычага: логика продукта, не техничка — ничего не потеряно,
ничего лишнего, порядок разумный; увидел кривизну — наложи вето. В режиме
«прототип» увидишь тест-задачи с пометкой «рекомендован» на хрупких местах:
оставь или вычеркни — но осознанно.

**Среда.** «Нарежь стори 01 на задачи». Постановщик пишет самодостаточные ТЗ
(что сделать, какой агент, арх-дельта) и коммитит их артефактом. Отдельного
гейта здесь нет — можешь заглянуть, но не обязан.

**Четверг.** «Возьми стори 01 в работу». Исполнитель берёт задачи по ТЗ,
коммитит после каждой, ведёт STATUS. Если ТЗ неполно или нарезка кривая —
Исполнитель возвращает это Постановщику, не тебе. Твоя работа — заглядывать в
`docs/STATUS.md`: секция «Требуется от PO» пуста — всё едет само.

**Пятница.** Стори на приёмке. Критерии уже отмечены Исполнителем с
доказательствами («запустил X → увидел Y»). Проверь руками один-два ключевых,
остальное — по доказательствам. Принял — берётся следующая.

### Что-то пошло не так

**Стори не принята.** Пиши замечания в чат как есть — Исполнитель превратит их в
задачи с пометкой «возврат» и снимет галочки *до* починки; нужна перенарезка —
уйдёт к Постановщику. Ничего не потеряется, даже когда чат исчезнет.

**Сессия оборвалась.** Ничего не делай. Следующая сессия начинается с
`git status`: агент сам увидит обрыв, доделает или откатит и запишет в стори,
что произошло. Просто скажи «продолжи стори N».

**Критичный баг.** Скажи прямо — баг встанет в STATUS («Требуется от PO») и
чинится мимо очереди. Некритичный — одна строка в BACKLOG.

### Эпик не достигнут (Гейт 3)

На ревью эпика ты можешь решить, что цель не достигнута. Тогда эпик переводится
в «возвращён (Гейт 3)», твои замечания Постановщик превращает в новые стори или
задачи, цикл повторяется. Это не провал процесса, а штатная петля.

### Прототип взлетел

Показал людям — и понял, что это надолго. Это событие, а не дрейф:
«Консультация: переключаем режим качества на продукт». Меняется строка в VISION,
первой стори следующего эпика Постановщик предложит «обложить тестами ядро».

### Вернулся к проекту спустя время

Открой `system/PAMYATKA.md` — она для этого и существует: три шага возврата,
команды ролей, что проверять на гейтах, словарик. Начало всегда одно:
`docs/STATUS.md` → секция «Требуется от PO» → «Продолжи стори N».

## Антипаттерны PO

- **Вбрасывать идеи в работающую стори** («а добавь заодно...»). Идея — в
  BACKLOG, стори не резиновая. Скоуп меняется только через гейты.
- **Принимать стори по коду.** Ты принимаешь поведение по критериям; код —
  территория агентов. Тянет заглянуть в код — почини критерии.
- **Не глядя пролистывать preview нарезки.** Дефолт — go, но взгляд стоит
  дёшево: здесь ошибки исправляются чтением, дальше — переделками. Критик уже
  вычитал форму; продуктовую логику видишь только ты.
- **Решать техничку за агента.** Забираешь ответственность, теряешь время.
  Твоя граница — то, что видит пользователь.
- **Копить открытые вопросы.** Полная секция вопросов = агент стоит.
- **Править VISION мимоходом.** Это конституция: меняется редко, осознанно,
  через консультацию.

| Файл | Пишет | Читает | Когда |
|---|---|---|---|

## Синхронизация правил из шаблона

Генерик-правила живут в `system/` и копируются в проект дословно; всё проектное —
в `PROJECT.md`. Когда флоу в шаблоне обновился: `Синхронизируй правила из шаблона`
— агент перезаписывает `system/` из шаблона **включая удаление файлов, которых
в шаблоне больше нет**, и обновляет штамп в `PROJECT.md`.
Улучшение, найденное в проекте, вносится В ШАБЛОН (через Методолога: ретро или
сессия методолога), потом синхронизируется вниз.

`.claude/` (политика команд, субагенты) и `CLAUDE.md` живут вне `system/` и
синхронизацией не едут: обновления доезжают до старых проектов руками или
аудитом Методолога.

## Перенос существующего проекта (например, прототипа)

1. Скопируй код в каталог кода (`src/` или свой — укажи в `PROJECT.md`).
2. Заполни `PROJECT.md` и `docs/VISION.md`.
3. Команда: «Проведи инвентаризацию кода: раздели на модули, создай паспорт для
   каждого, заполни ARCHITECTURE.md. Кода не менять» (Постановщик).
4. Проверь карту модулей — она станет фундаментом.
5. Дальше — обычный цикл с эпиков (первым часто стоит «приведение структуры к
   модульности», если прототип монолитный).
