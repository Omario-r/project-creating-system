# Шаблон проекта: агентная разработка (PO + агенты)

Универсальный каркас для разработки ПО с помощью ИИ-агентов, где человек выступает
в роли Product Owner. Клонируй эту папку в начале каждого нового проекта.

**Незнаком с системой?** Начни с `system/HOWTOUSE.md` — концепция за пять минут.

## Структура

```
CLAUDE.md              # Маршрутизатор ролей (тонкий; Claude Code читает сам)
PROJECT.md             # Оверлей проекта: стек, каталог кода, линтер (заполняет PO)
README.md              # Этот файл (инструкция для PO)
METHODOLOGY.md         # Журнал решений о процессе (читается только при смене флоу)
system/                # Генерик-правила — синхронизируются из шаблона, локально не править
  AGENT.md             #   роль Исполнителя (одно ТЗ → код)
  POSTANOVSHCHIK.md    #   роль Постановщика (эпик→стори, стори→ТЗ задач, арх-доки)
  ADVISOR.md           #   роль Советника (идея, Vision, эпик)
  DIZAINER.md          #   роль Дизайнера (мокапы экранов + DESIGN.md, канон визуала)
  METODOLOG.md         #   роль Методолога (доработка системы, ретро, аудит; читается только в своей сессии)
  ROSTER.md            #   каталог ролей × бэкендов, приёмка внешнего агента
  HANDOFF.md           #   контракт сдачи для внешнего агента (чек-лист + промпт)
  HOWTOUSE.md          #   концепция работы с шаблоном — точка входа для человека
  GUIDE.md             #   продолжение HOWTOUSE: сценарии, антипаттерны (для PO)
  templates/
    EPIC.md            #   шаблон эпика
    STORY.md           #   шаблон стори (ТЗ задач живут внутри)
    MODULE.md          #   шаблон паспорта модуля
    DESIGN.md          #   каркас визуальной конституции (→ docs/DESIGN.md)
    screen-mockup.html #   каркас HTML-мокапа экрана (→ docs/design/)
docs/                  # Продуктовые документы проекта
  VISION.md            #   конституция проекта
  ROADMAP.md           #   очередь эпиков + уроки
  STATUS.md            #   дашборд состояния (начинай отсюда)
  ARCHITECTURE.md      #   карта модулей
  DECISIONS.md         #   журнал ключевых решений
  BACKLOG.md           #   лоток идей и багов
  DESIGN.md            #   визуальная конституция: токены, индекс экранов (ведёт Дизайнер)
  design/<экран>.html  #   мокапы: один файл = один экран (канон визуала)
  epics/<имя-эпика>/EPIC.md + stories/01-<стори>.md
<каталог кода>/        # Код (каталог задаётся в PROJECT.md; по умолчанию src/)
  <модуль>/README.md   #   паспорт модуля + код
```

## Быстрый старт нового проекта

1. Скопируй папку шаблона, переименуй под проект.
2. Заполни `PROJECT.md` — стек, каталог кода, команда линтера, платформа.
3. Заполни `docs/VISION.md` — сам или в сессии-консультации
   («Прочитай `system/ADVISOR.md`. Помоги сформулировать Vision»).
4. Создай первый эпик — сам по шаблону или в консультации.
5. Эпик трогает видимое → «Дизайн-сессия по эпику <имя>» (Дизайнер): мокапы
   затронутых экранов + DESIGN.md, утверди каждый. Без этого декомпозиция
   не начнётся.
6. «Декомпозируй эпик <имя> в стори» (Постановщик) → утверди список (**Гейт 1**).
7. «Нарежь стори <N> на задачи» (Постановщик) → «Возьми стори <N> в работу»
   (Исполнитель). Дальше агенты сами.
8. Принимай стори по критериям (**Гейт 2**), отвечай на эскалации.
9. Все стори закрыты → ревью эпика (**Гейт 3**) → следующий эпик.

## Твой рабочий цикл (PO)

```
Vision → Эпик → [дизайн-сессия: мокапы экранов утверждены — если есть UI]
       → [Гейт 1: утверждение стори] → Постановщик режет, Исполнитель кодит
       → [ответы на «Открытые вопросы к PO»] → [Гейт 2: приёмка стори] × N
       → [Гейт 3: ревью эпика] → следующий эпик
```

Между гейтами система работает без тебя. Твои точки внимания:
- **Гейт 1** — 10 минут чтения экономят дни переделок. Проверяй логику продукта,
  не техничку: ничего не потеряно, ничего лишнего, порядок разумный.
- **Открытые вопросы** — отвечай быстро, они блокируют работу.
- **Гейт 2** — проверяй поведение по критериям приёмки, не код.
- **Гейт 3** — цель эпика достигнута? Нет — эпик возвращается, замечания станут
  новыми стори/задачами.
- **Идеи и баги** — не держи в голове: одна строка в `docs/BACKLOG.md`.

## Типовые команды агентам

> В Claude Code префикс «Прочитай X.md» не нужен: `CLAUDE.md` маршрутизирует сам
> по команде. Полные команды ниже — для любых других инструментов.

**Консультации (Советник, `system/ADVISOR.md`):**
- `Сырая идея: <две фразы>. Помоги понять, стоит ли делать.`
- `Помоги сформулировать Vision проекта.`
- `Хочу обсудить новый эпик: <идея в двух словах>.`

**Дизайн (Дизайнер, `system/DIZAINER.md`):**
- `Дизайн-сессия по эпику <имя>.` — экраны эпика → мокапы → утверждение.
- `Заведи DESIGN.md: обсудим визуальный подход проекта.`
- `Обнови мокап экрана <X>: <что меняем>.`

**Слой метода (Методолог, `system/METODOLOG.md`):**
- `Сессия методолога: <что обсуждаем в системе>.` — доработка правил/шаблонов.
- `Ретро по <эпику/обкатке>.` — уроки → шаблон → синхронизация.
- `Аудит проекта.` — гигиена процесса: протухшие доки, эпики-зомби, рассинхрон.

**Декомпозиция (Постановщик, `system/POSTANOVSHCHIK.md`):**
- `Декомпозируй эпик <имя> в стори.`
- `Нарежь стори docs/epics/<эпик>/stories/<файл> на задачи.`

**Исполнение (Исполнитель, `system/AGENT.md`).** Три пути запуска — подробно в
`system/GUIDE.md` («Как запускать исполнителя»):
- Отдельной сессией: `Возьми стори <файл> в работу: начни с первой задачи.` /
  `Продолжи стори <файл> со следующей задачи.`
- Делегировать субагенту (в сессии оркестратора): `Запусти ispolnitel на стори <N> T<M>.`
  Для сложного ТЗ сначала — `Запусти critic на T<M>` (холодная вычитка).
- `Ответы на открытые вопросы: <...>. Обнови стори и продолжай.`

## Синхронизация правил из шаблона

Генерик-правила живут в `system/` и копируются в проект дословно; всё проектное —
в `PROJECT.md`. Когда флоу в шаблоне обновился: `Синхронизируй правила из шаблона`
— агент перезаписывает `system/` из шаблона и обновляет штамп в `PROJECT.md`.
Улучшение, найденное в проекте, вносится В ШАБЛОН, потом синхронизируется вниз.

## Перенос существующего проекта (например, прототипа)

1. Скопируй код в каталог кода (`src/` или свой — укажи в `PROJECT.md`).
2. Заполни `PROJECT.md` и `docs/VISION.md`.
3. Команда: «Проведи инвентаризацию кода: раздели на модули, создай паспорт для
   каждого, заполни ARCHITECTURE.md. Кода не менять» (Постановщик).
4. Проверь карту модулей — она станет фундаментом.
5. Дальше — обычный цикл с эпиков (первым часто стоит «приведение структуры к
   модульности», если прототип монолитный).
