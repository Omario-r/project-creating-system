# ROSTER — каталог агентных ролей и бэкендов

> Генерик. Две оси: **роль** (что делается) × **бэкенд** (кто исполняет).
> Постановщик на каждой задаче ставит в ТЗ «рекомендованный агент» = роль + бэкенд.
> Конкретные модели/агенты проекта — в `.claude/agents/` и `PROJECT.md`.

## Ось 1 — Роли (функция)

| Роль | Тир | Что делает | Когда назначать |
|---|---|---|---|
| Советник | сильный | спарринг по идее/Vision/эпику | консультация |
| Дизайнер | сильный | мокапы экранов + DESIGN.md, конвенция-чек | дизайн-сессия эпика с видимым UI (прод-код не пишет) |
| Постановщик | сильный | эпик→стори, стори→ТЗ, арх-доки, назначение | декомпозиция (прод-код не пишет) |
| Исполнитель | средний | одно ТЗ → код + тесты + локальные доки | по умолчанию для задачи |
| Исполнитель (тяжёлая) | сильный | то же, но задача сложна по существу | мудрёный алгоритм/рефакторинг при полном ТЗ |
| Критик | дешёвый | холодная вычитка ТЗ вслепую → список пробелов | сложные ТЗ до запуска Исполнителя |

**Флаг «тяжёлая»** — атрибут задачи, не отдельная роль: тот же Исполнитель на
сильной модели. Ставь, когда задача сложна ПО СУЩЕСТВУ даже при полном ТЗ (не
путать с «ТЗ недописано» — это эскалация).

## Ось 2 — Бэкенды (кто исполняет роль)

- **Claude (Claude Code субагенты)** — по умолчанию. Тир → модель:
  Советник/Дизайнер/Постановщик/«тяжёлая» → **Opus**; Исполнитель → **Sonnet**;
  Критик → **Haiku**. Заведены в `.claude/agents/`. Исполнителя и Критика
  Постановщик/PO спавнит **явно, по команде** — не автопилот, гейты целы.
- **Внешние агенты** — для overflow (кончились лимиты Claude) и параллельной
  работы; при дешёвой приёмке (см. ниже) — ещё и экономия лимитов на рутине.
  Записи:
  - **Cursor Composer 2.5 fast** — средний тир, GUI (ручной хендофф).
  - **Gigacode** — слабый тир, **бесплатен**, GUI (ручной хендофф). Допуск
    узкий: только простые задачи Исполнителя при полном ТЗ; НЕ назначать на
    флаг «тяжёлая», хрупкий домен (деньги/данные/расчёты/парсинг) и задачи,
    где решение выбирается из альтернатив.

  Внешний сервис видит код проекта — для чувствительных кодовых баз внешние
  бэкенды не назначать. Список расширяется отдельной задачей (подбор +
  CLI-механизмы запуска).

## Хендофф внешнему агенту

Самодостаточное ТЗ (Фаза 1) = универсальный хендофф: задачу можно отдать любому
бэкенду. Пометь в ТЗ «бэкенд: <агент>». Механизм зависит от поверхности агента:

- **Claude-субагент** → нативный запуск через Task-инструмент.
- **Агент с CLI/API** → запуск через Bash.
- **GUI-агент (Cursor Composer, Gigacode)** → headless-запуска нет: PO копирует
  в чат агента готовый промпт-запуск из ТЗ задачи (канонический шаблон —
  `system/HANDOFF.md`), результат возвращается **коммитом по DoD**. Тот же
  контракт — любой бэкенд.

Внешний агент правил системы (`system/AGENT.md`, `PROJECT.md`) сам НЕ читает, а
длинный свод слабая модель не удержит. Поэтому его контракт — **`system/HANDOFF.md`**:
дистиллят DoD нумерованным чек-листом + готовый промпт-запуск для PO. Всё
проектно-конкретное (команды линтера/тестов, пути, сигнатуры) обязано лежать
прямо в ТЗ — требования к ТЗ для внешнего бэкенда см. `POSTANOVSHCHIK.md` § 4.

## Приёмка результата внешнего агента (три ступени)

Отчёту внешнего агента не верить; принимает **Постановщик** (операция
`прими результат задачи <N>`, см. `POSTANOVSHCHIK.md` § 4) ДО того,
как задача считается готовой. Ступени по возрастанию цены — дорогая почти
никогда не наступает:

1. **Ре-ран линтера и тестов** (Bash, токен-копейки) — независимо, не по claim'у
   агента. Плюс `git status`: коммит-транзакция полная, ничего не висит.
2. **Пост-критик на дешёвой модели**: Постановщик вставляет в промпт `critic`
   ТЗ + `git diff` коммита задачи; критик возвращает список пробелов против ТЗ
   и чек-листа `HANDOFF.md`. Дифф ≤300 строк (бюджет задачи) — операция дешёвая.
3. **Находки возвращаются внешнему агенту** одним сообщением-списком (он
   бесплатный — пусть чинит сам); текст-возврат готовит Постановщик, вставляет
   в чат агента PO. Доводка сильной моделью — только после двух неудачных
   возвратов; это же сигнал, что задача назначена не тому бэкенду.

Дорогой полный аудит главной сессией (как на первой обкатке) — не рабочий режим,
а разовая проверка нового бэкенда.
