# ROSTER — каталог агентных ролей и бэкендов

> Генерик. Две оси: **роль** (что делается) × **бэкенд** (кто исполняет).
> Постановщик на каждой задаче ставит в ТЗ «рекомендованный агент» = роль + бэкенд.
> Конкретные модели/агенты проекта — в `.claude/agents/` и `PROJECT.md`.

## Ось 1 — Роли (функция)

| Роль | Тир | Что делает | Когда назначать |
|---|---|---|---|
| Советник | сильный | спарринг по идее/Vision/эпику | консультация |
| Постановщик | сильный | эпик→стори, стори→ТЗ, арх-доки, назначение | декомпозиция (прод-код не пишет) |
| Исполнитель | средний | одно ТЗ → код + тесты + локальные доки | по умолчанию для задачи |
| Исполнитель (тяжёлая) | сильный | то же, но задача сложна по существу | мудрёный алгоритм/рефакторинг при полном ТЗ |
| Критик | дешёвый | холодная вычитка ТЗ вслепую → список пробелов | сложные ТЗ до запуска Исполнителя |

**Флаг «тяжёлая»** — атрибут задачи, не отдельная роль: тот же Исполнитель на
сильной модели. Ставь, когда задача сложна ПО СУЩЕСТВУ даже при полном ТЗ (не
путать с «ТЗ недописано» — это эскалация).

## Ось 2 — Бэкенды (кто исполняет роль)

- **Claude (Claude Code субагенты)** — по умолчанию. Тир → модель:
  Советник/Постановщик/«тяжёлая» → **Opus**; Исполнитель → **Sonnet**;
  Критик → **Haiku**. Заведены в `.claude/agents/`. Исполнителя и Критика
  Постановщик/PO спавнит **явно, по команде** — не автопилот, гейты целы.
- **Внешние агенты** — для overflow (кончились лимиты Claude), параллельной
  работы и экономии лимитов. Записи:
  - **Cursor Composer 2.5 fast** — средний тир, GUI (ручной хендофф).
  - **Gigacode** — слабый тир, **бесплатен**, GUI (ручной хендофф). Назначать
    только на простые задачи Исполнителя при полном ТЗ; НЕ назначать на флаг
    «тяжёлая» и хрупкий домен (деньги/данные/расчёты/парсинг). Бесплатность
    делает его дефолтным overflow-кодером для рутинных задач.

  Список расширяется отдельной задачей (подбор + CLI-механизмы запуска).

## Хендофф внешнему агенту

Самодостаточное ТЗ (Фаза 1) = универсальный хендофф: задачу можно отдать любому
бэкенду. Пометь в ТЗ «бэкенд: <агент>». Механизм зависит от поверхности агента:

- **Claude-субагент** → нативный запуск через Task-инструмент.
- **Агент с CLI/API** → запуск через Bash.
- **GUI-агент (Cursor Composer, Gigacode)** → headless-запуска нет: PO/оркестратор
  маршрутизирует ТЗ вручную (вставляет в агента), результат возвращается
  **коммитом по DoD** (`system/AGENT.md` §5, §7). Тот же контракт — любой бэкенд.

**Инлайн-DoD обязателен.** Внешний агент НЕ читает `system/AGENT.md` и
`PROJECT.md` — всё, что для Claude-бэкенда живёт в правилах, для внешнего должно
лежать прямо в ТЗ. Назначая внешний бэкенд, Постановщик добавляет в ТЗ блок
**«Сдача»** (шаблон — `system/templates/STORY.md`): один коммит-транзакция
(код + тесты + паспорта + файл стори + `STATUS.md`), новый тест по «Результату»,
только детерминированная верификация, формат сообщения коммита, не выходить за ТЗ.

## Приёмка результата внешнего агента

Результат внешнего (особенно слабого) агента принимает оркестратор/Постановщик
ДО того, как задача считается готовой — отчёту агента не верить:

1. **Независимый ре-ран** линтера и тестов (не полагаться на claim агента).
2. **Чек-лист «Сдачи»** по пунктам: коммит-транзакция полная? файл стори и
   `STATUS.md` обновлены? запрошенный тест добавлен?
3. **Читка диффа на корректность** — слабый агент пропускает связки (типовое:
   добавил поле, но не учёл его в сравнении/перерисовке/equals).

Найденные пробелы — доводка оркестратора отдельным коммитом
(`<эпик>/<стори> T<N> доводка: <суть>`) или возврат ТЗ агенту.
