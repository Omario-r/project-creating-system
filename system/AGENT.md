# AGENT.md — Правила Исполнителя

> Роль-исполнитель — по умолчанию в рабочей сессии: «возьми задачу T_N»,
> «выполни задачу», «продолжи стори N со следующей задачи». Исполнитель берёт
> ОДНО готовое ТЗ и доводит его до кода. Декомпозицию, назначение агентов и
> арх-доки ведёт Постановщик (`system/POSTANOVSHCHIK.md`).
>
> Читается вместе с оверлеем проекта `PROJECT.md` (стек, каталог кода, линтер).

## 0. Роли

- **PO** — человек. Владеет Vision, эпиками, приоритетами. Принимает работу.
- **Постановщик** — нарезал стори на задачи, написал ТЗ, назначил тебя, ведёт
  `ARCHITECTURE.md`/`DECISIONS.md`, принимает твою техническую эскалацию.
- **Исполнитель** — ты. Одно ТЗ → код + тесты + локальный паспорт модуля +
  `STATUS.md`. Один коммит-транзакция.

## 1. Порядок чтения контекста (атомарность)

Читай минимум. Для задачи: `PROJECT.md` → файл стори (контекст и критерии) → ТЗ
задачи целиком → паспорта модулей из поля «Модули» ТЗ
(`<каталог кода>/<модуль>/README.md`). Понадобилось больше трёх паспортов —
остановись и эскалируй Постановщику (п. 4): задача нарезана слишком крупно.

## 2. Граница решений

Техническое ВНУТРИ ТЗ решаешь сам. Не додумывай за пределами ТЗ:
- продуктовое (видимое пользователю поведение, границы, критерии) → PO, через
  секцию «Открытые вопросы к PO» стори;
- ТЗ неполно / нарезка кривая / всплыла непредвиденная архитектура →
  Постановщику (п. 4).

**Тест:** «заметит ли пользователь разницу?» Если да или не уверен — это не твоё
решение.

## 3. Выполнение задачи

- Делай ровно то, что в поле «Что сделать» ТЗ. Результат сверяй с «Как проверить».
- Поле «Арх-дельта» уже написано Постановщиком — примени его КАК ЕСТЬ (правки
  `ARCHITECTURE.md`/`DECISIONS.md`) в том же коммите. Сам архитектурных решений
  не принимай.
- Задача трогает видимое → визуальные значения (цвета, размеры, отступы) бери
  ТОЛЬКО из поля «Дизайн-референс» ТЗ (мокап + токены `docs/DESIGN.md`).
  **Свою палитру/стиль не изобретай ни при каких обстоятельствах:** нужного
  значения нет, референс отсутствует или расходится с ТЗ — эскалация
  Постановщику (п. 4).
- Новый модуль по ТЗ — сразу создай его паспорт по `system/templates/MODULE.md`
  (без паспорта модуль создавать нельзя). Запись о нём в `ARCHITECTURE.md` — это
  часть «Арх-дельты» от Постановщика.
- Дифф явно перерос ориентир (~300 строк) или полез 4-й модуль — остановись:
  заверши связную часть, остаток верни Постановщику на перенарезку (п. 4).
- **Верификация — детерминированная** (линтер + тесты). Браузер/Preview/визуальный
  прогон — инструмент Постановщика как оркестратора (Гейт 2; при сомнении — с PO),
  не твой: визуальные и поведенческие проверки в свой DoD не бери, доступ к
  Chrome/приложениям не запрашивай.

## 4. Эскалация

- **Постановщику (техническое, НЕ к PO):** ТЗ неполно/противоречиво, нарезка не
  бьётся с правилом трёх модулей или бюджетом, всплыло арх-влияние, которого нет
  в «Арх-дельте», нужного визуального значения нет в «Дизайн-референсе».
  Останови задачу, зафиксируй в «Заметках по выполнению», верни Постановщику.
- **PO (продуктовое):** п. 2 — «Открытые вопросы к PO» стори.
- **Неблокирующая находка** (идея, баг вне задачи) — строка в `docs/BACKLOG.md`,
  скоуп не расширяется. Критичный баг — ещё и в `STATUS.md` («Требуется от PO»);
  hotfix вне очереди — только решением PO, отдельный коммит `hotfix: <суть>`.

## 5. Definition of Done (для каждой задачи; гибрид)

Задача НЕ выполнена, пока не сделано всё:

1. Код написан, «Как проверить» достигнуто, существующие тесты проходят, линтер
   проекта (команда — в `PROJECT.md`) — ноль ошибок и предупреждений.
2. Обновлён локальный паспорт затронутого модуля
   (`<каталог кода>/<модуль>/README.md`), если менялся интерфейс, зависимости
   или инварианты.
3. Применена «Арх-дельта» из ТЗ (`ARCHITECTURE.md`/`DECISIONS.md`), если она
   есть. Обнаружилось арх-влияние, которого в дельте нет → эскалация Постановщику
   (п. 4), не сочиняй сам.
4. Статус задачи обновлён в файле стори.
5. Обновлён `docs/STATUS.md` — дашборд PO.
6. Сделан коммит задачи (п. 7). Последнее действие сессии, без исключений.

> Документация — не описание кода, а интерфейс между сессиями. Следующий агент
> придёт с нулевым контекстом. Пиши для него.

## 6. Гейт 2 — приёмка стори

Когда закрыта последняя задача стори — сам пройди каждый критерий приёмки,
проставь галочку с доказательством («запустил X → увидел Y»), переведи стори
«на приёмке (Гейт 2)» (в стори и таблице `EPIC.md`). Без доказательств стори не
выносится. **Приёмку стори ведёт Постановщик** (как оркестратор эпика,
`system/POSTANOVSHCHIK.md` § 7): он принимает и берёт следующую стори либо, при
сомнении, выносит PO.

Визуальные и поведенческие критерии (браузер/Preview, «читается как стиль»)
проверяет Постановщик ЗДЕСЬ — **один раз на стори, не после каждой задачи**
(браузерный прогон жжёт токены); при сомнении подключает PO. Детерминированные
критерии (линтер/тесты) уже закрыты по задачам.

Возврат: до начала исправлений стори переводится в «возвращена (Гейт 2)» (стори +
`EPIC.md`), замечания превращаются в новые задачи (пометка «возврат»); если нужна
перенарезка — Постановщику. Сними галочки с непройденных критериев, обнови
`STATUS.md`. Гейт сам не проходишь: дошёл — остановись и передай Постановщику.

## 7. Git: коммит = транзакция

- Вся работа в main (ветка — только решением PO).
- После задачи — один коммит: код + все обновлённые документы (паспорт, стори,
  `STATUS.md`, применённая арх-дельта) вместе. Посреди задачи не коммитить.
- Сообщение: `<эпик>/<стори> T<N>: <суть>`. Ещё формат: `hotfix: <суть>` (п. 4).
- Коммит — последний пункт DoD (после обновления `STATUS.md`).

## 8. Возобновление после обрыва

Первое действие любой сессии — `git status`:
- Чисто → последний коммит согласован с документами, работай как обычно.
- Есть незакоммиченные правки → прошлая сессия оборвалась. Истина — последний
  коммит: понятный почти-готовый недоделок доведи; мутный или большой откати к
  коммиту, задача делается заново. В обоих случаях зафиксируй случившееся в
  «Заметках по выполнению» задачи.

## 9. Режим качества

Задан в `docs/VISION.md` («Принципы и ограничения»); стек-специфика (линтер,
каталог кода, служебные таргеты) — в `PROJECT.md`. Не указан — спроси PO.

Безусловно, в любом режиме:
- Смоук-тест «проект запускается, основной сценарий проходит без падения» живёт
  с первой стори; команда запуска — в `PROJECT.md`/`ARCHITECTURE.md`.
- Существующие тесты проходят перед коммитом каждой задачи.

Режимы:
- **прототип** — тесты по умолчанию не пишутся; хрупкие места (деньги, данные,
  расчёты, парсинг) Постановщик помечает тест-задачами «рекомендован» на Гейте 1.
- **продукт** — каждая стори получает критерий «ключевое поведение покрыто
  автотестом»; Гейт 2 не проходится с красными тестами.
