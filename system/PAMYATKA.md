# PAMYATKA — шпаргалка PO

> Единственный PO-документ в живом проекте: вернулся спустя время — начни здесь.
> Полный документ системы (концепция, сценарии, антипаттерны) — README шаблона.
> Агентам не читать: правил здесь нет.

## Вернулся к проекту — три шага

1. Открой `docs/STATUS.md` — что в работе, последние результаты, следующий шаг.
2. Секция **«Требуется от PO»** — это твоя очередь: ответы на вопросы,
   утверждения, приёмка. Пока она не пуста — агенты стоят.
3. Продолжение — одно сообщение: `Продолжи стори <N>`. Обрыв прошлой сессии
   агент разрулит сам через `git status` — ничего не восстанавливай руками.

## Команды → роли

| Скажи | Проснётся | Получишь |
|---|---|---|
| `Консультация: <идея / Vision / эпик>` | Советник | спарринг, черновик Vision/эпика |
| `Дизайн-сессия по эпику <X>` | Дизайнер | HTML-мокапы + DESIGN.md на утверждение |
| `Декомпозируй эпик <X> в стори` | Постановщик | нарезка (вычитана Критиком) → preview тебе, дефолт go |
| `Нарежь стори <N> на задачи` | Постановщик | самодостаточные ТЗ (без гейта) |
| `Возьми стори <N> в работу` / `запусти ispolnitel на T<M>` | Исполнитель | код + коммиты + STATUS |
| `Прими результат задачи <N>` | Постановщик | 3-ступенчатая приёмка внешней задачи |
| `Ретро по <X>` / `Аудит проекта` / `Сессия методолога` | Методолог | уроки → шаблон; гигиена процесса |
| `Синхронизируй правила из шаблона` | — | свежий `system/` + штамп в PROJECT.md |

## Эпик ведёт Постановщик; ты принимаешь эпик

Постановщик рулит эпиком от нарезки до последней стори. Твоя обязательная
остановка — одна: **Гейт 3** (приёмка эпика). До запуска решаете открытые вопросы
вместе; в процессе Постановщик выныривает к тебе сам, когда есть сомнение.

- **Preview нарезки (не гейт).** Постановщик показывает список стори — скользни
  глазами: логика продукта, не техничка. **Дефолт — «поехали»:** молчишь = старт.
  Видишь потерю/лишнее/кривую границу — скажи (вето), иначе работа идёт. Нарезку
  до тебя уже вслепую вычитал Критик — это подстраховка, не замена твоему взгляду.
- **Гейт 2 — приёмка стори теперь у Постановщика.** Он сам проверяет критерии (в
  т.ч. визуал в браузере) и чейнит стори дальше. К тебе стори приходит только
  когда Постановщик сомневается и нужно твоё слово — тогда смотри поведение по
  критериям, замечания пиши как есть.
- **Гейт 3 — ревью эпика (твоё).** Цель достигнута? Нет — эпик возвращается,
  замечания станут новыми стори. Уроки — в ROADMAP.

**Ещё система ждёт от тебя:** ответы на «Открытые вопросы к PO» в стори
(блокируют работу — отвечай быстро); утверждение каждого мокапа (без этого эпик
с UI не режется); идеи и баги — не в голове, а строкой в `docs/BACKLOG.md`;
критичный баг — скажи прямо, починится мимо очереди.

## Словарик

- **Эпик** — крупная цель через ценность («игрок может X»); живёт в
  `docs/epics/<имя>/EPIC.md`, очередь — в ROADMAP.
- **Стори** — вертикальный кусок эпика, принимаемый по критериям на Гейте 2;
  внутри неё живут ТЗ задач.
- **ТЗ** — самодостаточное задание на одну задачу: что сделать, файлы, критерии,
  агент. Полное ТЗ — то, что позволяет отдать задачу любому исполнителю.
- **Гейт** — контрольная точка приёмки. Твой обязательный — только Гейт 3
  (приёмка эпика); Гейт 2 (приёмка стори) ведёт Постановщик, к тебе выносит при
  сомнении; preview нарезки — уведомление, не гейт.
- **DoD** (Definition of Done) — чек-лист готовности задачи: линтер ноль,
  тесты зелёные, паспорт и STATUS обновлены, коммит сделан.
- **Коммит-транзакция** — одна задача = ровно один коммит со всем, что она меняет.
- **Эскалация** — возврат работы вверх: техническая (Исполнитель → Постановщик,
  тебя не трогает) или продуктовая («Открытые вопросы к PO» — твоя).
- **Оверлей** — `PROJECT.md`: всё проектное (стек, линтер, карве-ауты) поверх
  генерик-правил `system/`, которые локально не правятся.
- **Синхронизация** — перезапись `system/` из шаблона (включая удаление файлов,
  которых там больше нет) + штамп в PROJECT.md.
- **Арх-дельта** — заранее вписанное в ТЗ изменение карты модулей; Исполнитель
  применяет как есть, не сочиняет.
- **Паспорт модуля** — README модуля в каталоге кода: зачем, границы, связи.
- **Режим качества** — «прототип» (без обязательных тестов, но со смоук-тестом)
  или «продукт» (каждая стори покрыта); задан в Vision, меняется консультацией.
- **Ростер** — каталог «роль × бэкенд» (`system/ROSTER.md`); Постановщик
  назначает агента каждой задаче.
- **Хендофф** — передача задачи внешнему агенту: промпт из ТЗ + чек-лист
  `system/HANDOFF.md`, результат — коммитом, потом приёмка.
- **Холодная вычитка** — дешёвый агент (`critic`) читает сложное ТЗ вслепую до
  запуска Исполнителя и возвращает список пробелов.
- **Мокап** — HTML-файл экрана в `docs/design/`, один файл = один экран;
  утверждённый мокап — канон визуала, агенты берут значения только из него.

## Куда смотреть

`docs/STATUS.md` — всегда первым; `docs/ROADMAP.md` — очередь эпиков и уроки;
`docs/BACKLOG.md` — куда кидать идеи; `docs/epics/…` — стори и критерии для
гейтов; `git log` — история читается задачами.
